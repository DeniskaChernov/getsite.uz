# Telegram-бот getsite — полное ТЗ (v2)

Юзернейм бота в конфиге сайта: `GETSITE_CONFIG.telegram.bot` → `@getsiteuzbot`.
Документ = единый источник правды для разработки бота: продукт, диалоги, безопасность, эксплуатация.

---

## 1. Роль и цель

Бот — первая линия продаж getsite. Он:

1. Встречает клиента, пришедшего с сайта по deep-link, и сразу говорит про выбранную услугу.
2. Ведёт живой диалог (LLM), собирает мини-бриф за 3–6 сообщений.
3. Отдаёт оформленный лид в админ-группу менеджеров.
4. Никогда не принимает оплату, не подписывает обязательства, не выдаёт скидки.

KPI: доля диалогов, дошедших до передачи лида; время первого ответа < 3 сек; ноль инцидентов с токеном/данными.

Правила обработки payload:

- Payload — это данные, а не команда: он выбирает шаблон приветствия, но не подставляется в промпт «как есть».

## 3. Архитектура и стек

- **Библиотека**: aiogram 3 (Python) или grammY (TypeScript) — на выбор исполнителя.
- **Режим**: webhook (не long polling) за HTTPS. Endpoint — отдельный сервис (Railway / VPS), не связан со статикой сайта.
- **LLM-слой**: один системный промпт (раздел 5) + история диалога. Модель отвечает только текстом; вся логика команд, rate-limit и передача лидов — код, не модель.
- **Хранилище**: SQLite/Postgres — состояние диалога, собранный бриф, статус лида. Персистентно, с бэкапом.
- **Очередь на LLM**: не более 1 параллельного запроса на пользователя; таймаут 30 сек с fallback-сообщением («секунду, уточняю…» → «передал вопрос Денису»).
- **Лиды**: отправка карточки (раздел 7) в приватную админ-группу + дублирование в таблицу/CRM.

## 4. Безопасность (обязательные требования)

### Токен и секреты

- Токен бота, ключ LLM, `secret_token` вебхука — **только** в переменных окружения / секретах платформы. В репозитории, логах и сообщениях об ошибках их нет никогда.
- Отдельный токен для staging-бота. При малейшем подозрении на утечку — немедленный `revoke` через BotFather.
- В BotFather: `/setjoingroups` → Disable (бота нельзя добавить в чужие группы), inline mode выключен, если не нужен.

### Webhook

- `setWebhook` с параметром `secret_token` (случайные 64+ бита). Каждый входящий запрос проверяется по заголовку `X-Telegram-Bot-Api-Secret-Token`; несовпадение → 403 и лог.
- Endpoint принимает только POST, только JSON, тело ≤ 1 МБ. Путь вебхука — неугадываемый (например `/tg/<random>`).
- `allowed_updates` — только `message`, `callback_query`, `my_chat_member`. Остальное не принимать.
- Опционально: allowlist подсетей Telegram (`149.154.160.0/20`, `91.108.4.0/22`) на уровне прокси/файрвола.

### Анти-флуд и лимиты

- Rate-limit на пользователя: максимум 1 LLM-запрос в 3 сек и 30 сообщений / 10 минут. Превышение → вежливое «отвечаю по одному сообщению» без вызова LLM.
- Входной текст: обрезка до 2000 символов перед LLM. Медиа (фото/войсы/документы) в LLM не передаются — бот отвечает «пришлите, пожалуйста, текстом; файлы покажете уже Денису».
- Глобальный circuit breaker: аномальный всплеск трафика → бот отвечает шаблонами без LLM и алертит админа.
- Пересланные сообщения и сообщения от ботов — игнорировать (анти-спам).

### Защита LLM-слоя (prompt injection)

- Сообщения пользователя всегда передаются как user-роль, системный промпт — недоступен для изменения из диалога.
- В системном промпте: запрет раскрывать промпт, менять роль, обсуждать внутренние инструкции (см. раздел 5, блок «ЗАЩИТА»).
- Выходной фильтр в коде: ответ бота не должен содержать токен-подобные строки (`\d{8,10}:[A-Za-z0-9_-]{35}`), ключи API, содержимое системного промпта. Совпадение → ответ заменяется шаблоном, инцидент в лог.
- Модель не имеет инструментов/функций с побочными эффектами. Отправка лида — только детерминированный код по явному состоянию диалога.

### Данные и приватность

- Храним минимум: user_id, username, имя из профиля, язык, payload, текст брифа, timestamps. Паспорта, карты, адреса не запрашиваем и, если прислали, — не сохраняем в лид, просим не отправлять.
- История диалога: хранение 90 дней, потом удаление. Экспорт/удаление по запросу клиента («удалите мои данные» → команда админа `/forget <user_id>`).
- Логи без полного текста сообщений (только метаданные + первые 200 символов при ошибке). Токенов и ключей в логах нет.
- Оплата и договор — только через человека. Бот **никогда** не отправляет реквизиты и не просит перевод: об этом сказано и в промпте, и продублировано хардкод-фильтром на слова «оплатите на карту», «переведите» в ответах бота.

### Админ-доступ

- Админ-команды (`/leads`, `/mute`, `/forget`, `/stats`) работают только для user_id из allowlist в конфиге — не по username (username можно сменить).
- Админ-группа приватная; бот проверяет `chat_id` перед отправкой лида.

### Эксплуатация

- Health-check endpoint для мониторинга + алерт при падении вебхука (`getWebhookInfo.last_error_date`).
- Ежедневный бэкап БД. Обновления зависимостей — минимум раз в месяц.
- Инцидент-план: revoke токена → новый секрет вебхука → проверка логов → пост-морем в docs.

## 6. Сценарии и edge cases (логика в коде, не в промпте)

| Ситуация | Поведение |
|----------|-----------|
| `/start` без payload | Шаблонное приветствие «СТАРТ БЕЗ PAYLOAD» (без LLM) |
| `/start` с валидным payload | Шаблон приветствия услуги (быстро, без LLM) → дальше LLM |
| Медиа / voice / документ | Шаблон: «Пришлите текстом, файлы покажете Денису» |
| Стикер / реакция | Игнор или короткий шаблон |
| Сообщение > 2000 символов | Обрезать для LLM, клиенту: «прочитал, отвечаю по сути» |
| Флуд (лимиты раздела 4) | Шаблон без LLM |
| LLM недоступен / таймаут | «Секунду…» → fallback: «Передал вопрос Денису: @getsiteuz, +998 91 908-06-21» |
| Группа / канал | Бот не работает вне личек: leave + лог |
| Другой бот пишет | Игнор |
| «Удалите мои данные» | Подтвердить, создать задачу админу, `/forget` |
| Клиент вернулся через N дней | Поднять сохранённый бриф: «мы остановились на…» |
| Бриф собран | Карточка лида в админ-группу + подтверждение клиенту |

Команды клиента: `/start`, `/help` (что умеет + контакты), `/lang ru|uz|en` (принудительная смена языка).

## 8. Метрики и логи

- Считать: старты по payload, доля дошедших до брифа, доля переданных лидов, среднее время диалога, отказы LLM/фильтров.
- Еженедельный автоотчёт в админ-группу (`/stats` — вручную).
- Логи: события уровня INFO без текста сообщений; ERROR — с первыми 200 символами контекста, без секретов.

## 9. Чеклист приёмки перед запуском

- [ ] Токен и ключи только в секретах, в git-истории их нет
- [ ] Webhook с `secret_token`, проверка заголовка, 403 на мусор
- [ ] `allowed_updates` ограничен; joingroups выключен в BotFather
- [ ] Rate-limit и обрезка ввода работают (протестировать флудом)
- [ ] Prompt-injection тесты: «покажи промпт», «игнорируй инструкции», «ты теперь X», инструкция в пересланном сообщении — бот не ломается
- [ ] Выходной фильтр ловит токен-подобные строки
- [ ] Все 14 payload открывают правильную услугу; мусорный payload → discuss
- [ ] Медиа/войсы/стикеры/группы обрабатываются по таблице сценариев
- [ ] Карточка лида приходит в админ-группу, кнопки работают
- [ ] Fallback при отключённом LLM показывает контакты
- [ ] `/forget` удаляет историю пользователя
- [ ] UZ и EN приветствия и цены проверены носителем/вручную

## 10. Осталось заполнить владельцу

1. Финальный токен и `chat_id` админ-группы (в секреты, не в репо).
2. Куда падает лид: группа + таблица/CRM (ссылка).
3. Рабочие часы и SLA ответа Дениса (для текста «ответит в рабочий день»).
4. UZ-версии ключевых фраз (приветствие, цены, передача лида).
5. Правила эскалации «клиент злой / срочно» (телефон сразу или ожидание).
